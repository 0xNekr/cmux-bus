# Multi-Agent Bus ‚Äî Protocol v1

This protocol coordinates two or more agents (e.g. Claude Code, Codex CLI)
running in adjacent [cmux](https://cmux.com) panes. Communication uses an
append-only event-sourced log per workspace.

## Files (per workspace)

- `.agents/bus.jsonl` ‚Äî append-only event log, one JSON object per line
- `.agents/agents.json` ‚Äî `{agent_name: cmux_surface_id}` registry
- `.agents/PROTOCOL.md` ‚Äî a copy of this document, generated by `agent-init`
- `AGENTS.md` (workspace root) ‚Äî short pointer that Codex CLI reads natively

## Event schema

```json
{
  "id":             "8-char id",
  "ts":             "ISO-8601 UTC",
  "from":           "agent name",
  "to":             "agent name (or 'user')",
  "type":           "ask|handoff|done|block|ack",
  "ref":            "id of parent event, or null",
  "status":         "open|in_progress|done|blocked",
  "paths_claimed":  ["glob", ...],
  "body":           "free text"
}
```

The bus is **append-only**: `status` is *declared* by an event, not mutated
on prior events. The effective state of a thread is the **last** event whose
`ref` chains back to the root.

Writers must append exactly one valid JSON object per line. Writes are
serialized with `.agents/bus.lock` so concurrent agents cannot interleave
partial JSON into `bus.jsonl`.

## Types

- `ask` ‚Äî open a thread asking the recipient for input
- `handoff` ‚Äî pass a task; recipient should `ack` then `done`
- `ack` ‚Äî acknowledge receipt; declares `in_progress`
- `done` ‚Äî close a thread with the result
- `block` ‚Äî escalate to user when stuck

## Routing defaults

These are conventions, not enforced rules:

- **Claude** ‚Äî design, critique, broad exploration, multi-file refactor, UX/API framing
- **Codex** ‚Äî CLI diagnostics, one-shot scripts, tests, bisect, migrations, hypothesis verification

## Inbox routine

Before starting any new task, every agent **must**:

1. Read `.agents/bus.jsonl` (if it exists)
2. Group events by thread root (`ref ?? id`)
3. For each thread where the last event targets you (`to == me`) and the
   declared status is not `done` or `blocked`: process it.
4. Once caught up, take new work.

Use `agent-inbox` (provided) to get this view automatically. Threads opened
by an agent whose surface is no longer registered appear with `[stale]`.

## Path ownership

When sending a `handoff`, declare `paths_claimed` (array of file paths or
globs). The other agent **must not** edit those paths until the thread is
closed by a `done` or `block` event.

This is a coordination convention, not a filesystem lock. Trust the
protocol.

## Disagreement rule

If a thread exchanges two `ask` ‚Üî `ask` round-trips between agents without
converging, the next event must be a `block` directed to `user`, with:

- the body summarizing the disagreement
- each agent's recommendation (declared in prior events)
- explicit options the user can choose

No agent edits the disputed paths until the user resolves the block.

## Signaling

After appending an event, the sender notifies the recipient by injecting
text into their cmux pane:

```
cmux send --surface <recipient_surface_id> "new <type> id=<event_id> from=<sender>"
cmux send-key --surface <recipient_surface_id> Enter
```

The recipient is expected to read the bus on signal. The signal is a
wake-up; the bus is the source of truth.

## Surface lifecycle

`agent-init` purges entries in `agents.json` whose `cmux_surface_id` no
longer corresponds to a live terminal surface in the current workspace
(detected via `cmux --id-format both surface-health`).

`agent-send` refuses to send to a recipient whose surface is no longer
live, with a clear error message asking the peer to re-run `agent-init`.

`agent-inbox` displays `[stale]` on threads whose `from` is no longer in
`agents.json`. It is read-only ‚Äî the bus is never mutated.

## Stuck detection

`agent-inbox` flags a thread as `[stuck Xm]` when the **last event**
declares `status: in_progress` and is older than the stuck threshold
(default 10 minutes). This is a heuristic for "the peer probably crashed
mid-task or the work was abandoned."

- Threshold is configurable per call: `--stuck-after MIN`
- Or globally via env: `AGENT_BUS_STUCK_AFTER_MIN`
- Setting it to `0` or a negative number disables the check
- Filters: `--no-stuck` hides stuck threads, `--only-stuck` shows only them
- JSON output adds `stuck: bool` and `age_minutes: N` per thread

Stuck detection only fires on `in_progress` ‚Äî an `open` thread is not
stuck, it's just unclaimed work, regardless of age.

## Recovery

When a peer crashes or a thread gets abandoned, two wrappers help close
or restart the work cleanly without breaking the append-only invariant:

- **`agent-cancel <id> [reason]`** appends a `block` event addressed to
  `user`, declaring `status: blocked`. This closes the thread cleanly.
  Refuses if the thread is already `done` or `blocked` (use `--force` to
  override).

- **`agent-resume <id> [body]`** appends a new `handoff` event referencing
  the thread root, addressed to the **original recipient** (the `to` of
  the root event). This re-opens the thread with the latest event being
  a fresh handoff, and sends a wake-up signal. Default body is
  `RESUME: <previous root body>`. Refuses if the thread is already `done`
  or `blocked` (use `--force` to override).

Both commands resolve the thread root from any event id in the chain ‚Äî
you can pass the root id, an ack id, anything that belongs to the
thread.

Manual cleanup of an arbitrary thread is also possible with
`agent-done <id>` (declares `done` from your own perspective). Choose:

- `agent-done` if **you** are completing or accepting the close
- `agent-cancel` if **the work is dropped** and should be escalated/audited
- `agent-resume` if **the work should be retried** by the original peer
